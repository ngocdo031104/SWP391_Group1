2.14. View Guide Dashboard

Primary Actors: Guide
Secondary Actors: None

Description: Allows tour guides to view their assigned tour schedules, participant lists with check-in status, incident reports, and operation logs.

Preconditions: 1. Guide is authenticated and logged in. 2. Guide has at least one tour assignment in the system.

Postconditions: 1. The system renders the Guide Dashboard showing all assigned tours with departure dates, destinations, and operational status. 2. Guide can navigate to sub-pages: Participant List, Incidents, Operation Logs.

Normal Sequence/Flow

  1. Guide accesses /guide/dashboard.
  2. System checks session: if not logged in → redirect to /login.
  3. System verifies role equals "Guide": if not → redirect to /home.
  4. System calls GuideDAO.getAssignmentsByGuideId(guideId) to fetch all TourAssignment records with JOINed TourSchedule, Tour, and TourStatus data.
  5. System forwards to /views/guide/dashboard.jsp.
  6. The JSP renders a table with columns: Tour Name, Destination, Departure Date, Return Date, Operational Status (Preparing/Scheduled/InProgress/Completed/Cancelled), and Action buttons.
  7. Each row displays four action buttons:
    • "Danh sách đoàn" → navigates to ?action=participants&scheduleId=X
    • "Sự cố" → navigates to ?action=incidents&scheduleId=X
    • "Nhật ký" → navigates to ?action=operationLogs&scheduleId=X
    • "Trạng thái" → opens status update modal

2.15. Manage Favorite Tours

Primary Actors: Customer
Secondary Actors: None

Description: Allows customers to save tours to their favorites list and view or remove saved tours.

Preconditions: Customer is authenticated and logged in.

Postconditions: 1. The tour is added to or removed from the customer's FavoriteTour list. 2. The heart icon on the tour card updates to reflect the current state.

Normal Sequence/Flow

  1. Customer views a tour card on the Home Page or Tour Discovery page.
  2. Customer clicks the heart (favorite) icon on the tour card.
  3. Client-side JavaScript sends AJAX POST to /customer/wishlist/toggle?tourId=X.
  4. WishlistController receives the request.
  5. System checks that the user is authenticated (sessionUser exists). If not logged in → returns HTTP 401 with JSON error message.
  6. WishlistDAO.toggleFavorite(userId, tourId) is called:
    • If the tour already exists in FavoriteTour → DELETE the record.
    • If the tour does not exist → INSERT a new FavoriteTour record.
  7. System returns JSON response: {"status": "added"} or {"status": "removed"}.
  8. Client-side JS updates the heart icon visual state.
  9. Customer can view all favorites at /customer/wishlist.
  10. WishlistViewController calls WishlistDAO.getFavoriteTours(userId) and forwards to wishlist.jsp.

2.16. Submit Tour Review

Primary Actors: Customer
Secondary Actors: None

Description: Allows customers to submit ratings and comments after completing a tour. The review form is embedded at the bottom of the Tour Detail page.

Preconditions: 1. Customer is authenticated and logged in. 2. Customer is viewing a tour detail page.

Postconditions: 1. A new Review record is inserted into the database with the customer's rating and content. 2. The page reloads via Post-Redirect-Get pattern and the new review appears in the review list.

Normal Sequence/Flow

  1. Customer scrolls to the review section on detail.jsp.
  2. Customer clicks star icons (1-5) to select rating; client-side JS updates hidden input #review-rating-input with the selected value. Default is 5 stars.
  3. Customer types review content in the textarea.
  4. Customer optionally clicks "Tải ảnh lên" (simulated — assigns a placeholder Unsplash image URL).
  5. Customer clicks "Gửi Đánh Giá" to submit the form.
  6. Form POSTs to /detail (DetailController.doPost()).
  7. System reads name and email from the SESSION user object (NOT from form fields — form fields are readonly and ignored for security).
  8. System reads content, rating, and tourId from request parameters.
  9. TourDAO.insertReview(name, email, tourId, rating, content) executes:
    • Step 1: Lookup UserID by email in [User] table.
    • Step 2: If user not found → create a new user account (roleId=4) with hashed password and a default UserProfile avatar.
    • Step 3: Find a BookingID where this user booked this tour (no status filter — any status accepted).
    • Step 4: If no booking found → use any booking for this tour as fallback.
    • Step 5: If still no booking → create a dummy booking with status='Completed' (BookingCode = "BK_DUMMY_xxx").
    • Step 6: INSERT INTO Review (TourID, BookingID, CustomerID, Rating, Content, IsVisible=1, ImageURL).
  10. System redirects to /detail?id={tourId} using Post-Redirect-Get pattern.
  11. Page reloads and the new review is displayed in the review list.

2.28. Assign Tour Guide

Primary Actors: Admin, Staff
Secondary Actors: None

Description: Allows administrators and staff to assign a tour guide to a specific departure schedule when creating or editing a schedule.

Preconditions: 1. Admin or Staff is authenticated. 2. The tour schedule exists or is being created. 3. At least one guide account exists in the system.

Postconditions: 1. The TourSchedule record is updated with the assigned GuideID. 2. A TourStatus history record is created with status 'Scheduled'. 3. A TourAssignment record is created or updated (MERGE upsert).

Normal Sequence/Flow

  1. Admin/Staff accesses /admin/schedules-pricing for a specific tour.
  2. Admin/Staff creates a new schedule or edits an existing one.
  3. In the schedule form, Admin/Staff selects a guide from the dropdown and optionally enters assignment notes.
  4. Form is submitted to AdminSchedulePricingController.doPost().
  5. If action is "addSchedule":
    • scheduleDAO.insertSchedule(sched) inserts the new TourSchedule with availableSeats = totalSeats.
    • If guideId > 0: guideDAO.assignGuideToSchedule(newId, guideId, assignedBy, notes) is called as a transaction:
  6. Transaction steps:
    • Step 1: UPDATE TourSchedule SET GuideID = ? WHERE ScheduleID = ?
    • Step 2: INSERT INTO TourStatus (ScheduleID, Status='Scheduled', Notes, UpdatedBy)
    • Step 3: MERGE INTO TourAssignment (upsert by ScheduleID + GuideID)
    • All three steps are committed or rolled back together.
  7. If action is "updateSchedule": system compares old GuideID with new GuideID. If changed (or old was null): calls guideDAO.assignGuideToSchedule().
  8. Admin/Staff can view assignment history at /admin/assignments.
  9. AdminAssignmentController.doGet() calls GuideDAO.getAllAssignments() and forwards to assignments.jsp.

2.29. Check In Participants

Primary Actors: Guide
Secondary Actors: None

Description: Allows tour guides to check in customers before a tour departs, tracking attendance status and confirmation timestamps in real-time.

Preconditions: 1. Guide is authenticated and logged in. 2. Guide is assigned to the schedule being managed. 3. The schedule has confirmed participants (BookingParticipant records with Booking status in Confirmed/Completed/Success).

Postconditions: 1. An Attendance record is created or updated (MERGE upsert) with the participant's check-in status. 2. The progress bar and participant status cells are updated on the UI without page reload.

Normal Sequence/Flow

  1. Guide clicks "Danh sách đoàn" on the dashboard for a specific schedule.
  2. System calls AttendanceDAO.getAttendanceByScheduleId(scheduleId) which LEFT JOINs BookingParticipant with Attendance and filters bookings with status != 'Cancelled'.
  3. System calculates checkedInCount and totalCount from the result set.
  4. System forwards to /views/guide/participants.jsp.
  5. JSP renders: Tour info grid, Progress bar (X/Y khách hàng with green fill), Table with columns: STT, Họ Tên, Loại Khách, Liên Hệ, Trạng Thái, Ghi Chú, Hành Động.
  6. Status column shows "Đã check-in (HH:mm dd/MM)" in green or "Chưa điểm danh" in gray.
  7. Guide clicks the toggle button for a participant.
  8. Client-side JS toggleCheckIn(participantId) reads current data-checked attribute, toggles it, reads notes, sends AJAX POST to /guide/dashboard with params: action=checkin, scheduleId, participantId, checkedIn, notes.
  9. GuideDashboardController.doPost() validates session, role=Guide, parses scheduleId, checks if guide is assigned.
  10. Calls AttendanceDAO.saveAttendance(scheduleId, participantId, checkedIn, guideId, notes):
    • MERGE INTO Attendance: if record exists → UPDATE CheckedIn, CheckInTime, CheckedBy, Notes; if not → INSERT.
    • If checkedIn=true → CheckInTime = current timestamp.
    • If checkedIn=false → CheckInTime = null.
  11. Returns JSON: {"status": "success", "checkInTime": "HH:mm dd/MM/yyyy", "message": "..."}.
  12. Client-side JS updates UI without reload: changes button text/color, updates status cell, calls updateProgress().

2.38. Export Revenue Reports

Primary Actors: Accountant, Admin
Secondary Actors: None

Description: Allows administrators and accountants to view analytics data and export reports as CSV files.

Preconditions: User is authenticated with role Admin (1) or Accountant (5).

Postconditions: 1. Analytics data is displayed in tabbed views. 2. CSV file is downloaded to the user's device.

Normal Sequence/Flow

  1. User accesses /admin/analytics.
  2. AdminAnalyticsController.doGet() checks session user has roleId=1 or roleId=5; otherwise redirects to /login.
  3. System forwards to /admin/analytics.jsp which renders 5 tabs: Revenue, Bookings, Performance, Guides, Reports.
  4. Each tab loads data via AJAX (?ajax=true&type=revenue|bookings|performance|guides|reports).
  5. Revenue tab: monthly revenue chart (getRevenueByMonth), revenue by category (getRevenueByCategory), revenue by top 10 tours (getRevenueByTour).
  6. Bookings tab: booking status distribution (getBookingStatusDistribution), booking trends over 30 days (getBookingTrends).
  7. Performance tab: tour performance with bookings, revenue, avg rating, occupancy rate (getTourPerformanceList).
  8. Guides tab: guide activity summary with experience, rating, tours led (getGuideActivitySummary).
  9. Reports tab: saved snapshot reports from AnalyticsReport table (getAllSavedReports).
  10. User clicks the export button on any tab.
  11. Client-side JavaScript confirmExport(tableId, filename) reads the HTML table content and generates a CSV file. The CSV is downloaded via a Blob URL — export is entirely client-side.
  12. Admin can save a data snapshot via POST action=saveSnapshot which inserts an AnalyticsReport record.

2.39. Perform Predictive Analytics

Primary Actors: Admin
Secondary Actors: None

Description: Allows administrators to run predictive models on historical booking and revenue data to generate forecasts for future periods.

Preconditions: 1. Admin is authenticated with roleId=1. 2. Sufficient historical data exists.

Postconditions: 1. Forecast data is generated and displayed. 2. A PredictionResult snapshot is saved to the database.

Normal Sequence/Flow

  1. Admin accesses /admin/forecast.
  2. AdminForecastController.doGet() checks session user has roleId=1; otherwise redirects.
  3. If not AJAX: system loads prediction history via AnalyticsDAO.getAllPredictions() and forwards to /admin/forecast.jsp.
  4. Admin selects forecast type: Revenue, BookingTrend, or Demand.
  5. Client-side JS sends AJAX GET to /admin/forecast?ajax=true&action=getCalculations&type={type}.
  6. AnalyticsDAO.calculateForecast(type) executes:

For Revenue or BookingTrend:

  • Fetches 6 months of historical data.
  • If empty → generates fallback data with zero values for 6 months.
  • Performs Simple Linear Regression: calculates slope and intercept.
  • Calculates R² as confidence score. If all zero → confidence = 85.0.
  • Generates predictions for next 3 months.
  • Returns: {historical, forecast, regression: {slope_a, intercept_b, r2}, confidence}.

For Demand:

  • Fetches TOP 5 tours by booking count in last 3 months with occupancy rates.
  • If no results → fallback: selects TOP 5 tours with random booking values (10-30) and occupancy rates (70-90%).
  • Calculates projected demand: round(avgBookingsPerMonth * 1.15 + 2).
  • Confidence = 88.5 (hardcoded).
  • Returns: {historical_top_tours, projected_demand, confidence: 88.5}.
  1. Chart.js renders historical vs forecast data.
  2. Admin clicks "Run Forecast" → POST to /admin/forecast. Creates PredictionResult with modelVersion="v1.0-Regression/WMA". Calls AnalyticsDAO.insertPrediction(pr).

2.42. Update Tour Status

Primary Actors: Guide
Secondary Actors: None

Description: Allows tour guides to update the operational status of a tour during execution (Preparing, Scheduled, InProgress, Completed, Cancelled).

Preconditions: 1. Guide is authenticated and logged in. 2. Guide is assigned to the schedule being updated.

Postconditions: 1. TourSchedule.Status is updated. 2. A TourStatus history record is inserted. 3. A TourOperationLog record is inserted.

Normal Sequence/Flow

  1. Guide clicks "Trạng thái" button on the dashboard for a specific schedule.
  2. Client-side JS openStatusModal(scheduleId, currentStatus) opens a modal dialog.
  3. Modal displays: hidden scheduleId input, dropdown (Preparing/Scheduled/InProgress/Completed/Cancelled) pre-selected to current status, notes textarea, "Xác nhận" and "Hủy bỏ" buttons.
  4. Guide selects new status and optionally enters notes, clicks "Xác nhận".
  5. JS submitStatusUpdate() sends AJAX POST to /guide/dashboard with params: action=updateStatus, scheduleId, newStatus, notes.
  6. GuideDashboardController.doPost() validates session, role=Guide.
  7. Verifies guide is assigned to this schedule (getAssignmentsByGuideId()). If not → 403.
  8. Validates newStatus ∈ {Preparing, Scheduled, InProgress, Completed, Cancelled}.
  9. Calls guideDAO.updateTourStatus(scheduleId, newStatus, notes, guideId) as transaction:
    • Step 1: UPDATE TourSchedule SET Status = newStatus.
    • Step 2: INSERT INTO TourStatus (ScheduleID, Status, Notes, UpdatedBy).
    • Step 3: INSERT INTO TourOperationLog (ScheduleID, Activity="HDV Cập nhật trạng thái thành: " + newStatus, OperatedBy).
    • All three committed or rolled back.
  10. Returns JSON {"status": "success"}.
  11. JS shows alert, closes modal, reloads page via window.location.reload().

2.43. Report Tour Incident

Primary Actors: Guide
Secondary Actors: None

Description: Allows recording and reporting of incidents that occur during a tour. Incidents are created with status 'Open' and can be viewed by severity level.

Preconditions: 1. Guide is authenticated and logged in. 2. Guide is assigned to the schedule.

Postconditions: 1. An IncidentReport record is created with status 'Open'. 2. The incident appears in the incident list.

Normal Sequence/Flow — Viewing

  1. Guide clicks "Sự cố" on the dashboard → GET /guide/dashboard?action=incidents&scheduleId=X.
  2. Controller verifies guide is assigned. If not → sets error, forwards to dashboard.
  3. IncidentReportDAO.getIncidentsByScheduleId(scheduleId) fetches incidents with reporter name via JOIN User, ordered by CreatedAt DESC.
  4. Forwards to /views/guide/incidents.jsp. Renders Tour info grid, "Báo Cáo Sự Cố Mới" button, table: Mã sự cố, Tiêu Đề, Thời Gian, Mức Độ (Low/Medium/High/Critical), Trạng Thái (Open/InProgress/Resolved/Closed), Mô Tả.

Normal Sequence/Flow — Reporting

  1. Guide clicks "Báo Cáo Sự Cố Mới" → JS openIncidentModal() opens modal with: title input, severity dropdown (Low/Medium/High/Critical, default Medium), description textarea.
  2. Guide fills form, clicks "Gửi báo cáo". JS validates title & description not empty.
  3. JS sends AJAX POST to /guide/dashboard with: action=reportIncident, scheduleId, title, severity, description.
  4. Controller validates session, role=Guide, verifies guide is assigned, validates severity ∈ {Low, Medium, High, Critical}.
  5. Creates IncidentReport object, calls IncidentReportDAO.insertIncidentReport(report): INSERT INTO IncidentReport (Status='Open').
  6. Returns JSON {"status": "success"}. JS alerts, closes modal, reloads page.

2.44. View Tour Operation Logs

Primary Actors: Guide, Admin
Secondary Actors: None

Description: Allows guides to view operation logs for their assigned schedules in a timeline format, and allows admins to view all operation logs across all schedules with search and pagination.

Preconditions: 1. User is authenticated. 2. Guide: must be assigned to the requested schedule. 3. Admin/Staff: must have Admin or Staff role.

Postconditions: 1. Guide sees a timeline view of all operational activities for one schedule. 2. Admin sees a paginated table of all operation logs with search capability.

Normal Sequence/Flow — Guide View

  1. Guide clicks "Nhật ký" → GET /guide/dashboard?action=operationLogs&scheduleId=X.
  2. Controller verifies guide is assigned. If not → error, forwards to dashboard.
  3. TourOperationLogDAO.getLogsByScheduleId(scheduleId) fetches logs JOINed with User (operator name) and Role (operator role), ordered by CreatedAt DESC.
  4. Forwards to /views/guide/operation-logs.jsp. Renders Tour info grid + Timeline view: vertical line, circular markers (first green, rest blue).
  5. Each timeline item: timestamp (HH:mm:ss - dd/MM/yyyy), activity description, operator name with role badge.

Normal Sequence/Flow — Admin View

  1. Admin clicks "Nhật Ký Vận Hành" in sidebar → GET /admin/operation-logs?search=&page=1.
  2. AdminOperationLogController verifies role is Admin or Staff.
  3. Calls TourOperationLogDAO.getAllLogsPaged(page, 15, search) with pagination (15 rows/page) and search across Activity, FullName, TourName.
  4. Calls TourOperationLogDAO.getLogsCount(search) → calculates totalPages.
  5. Forwards to /admin/operation-logs.jsp. Table: Mã Log, Tour & Lịch KH, Hoạt Động, Người TH, Vai Trò, Thời Gian + Search bar + Pagination.

2.46. Moderate User Content

Primary Actors: Admin
Secondary Actors: None

Description: Allows administrators to moderate reviews, community posts, and comments by hiding inappropriate content, restoring hidden content, managing flagged reports from users, and viewing moderation history.

Preconditions: Admin is authenticated with roleId=1 or has "Content Management" permission.

Postconditions: 1. Content visibility (IsVisible) is toggled. 2. A ModerationRecord is created. 3. Flagged status (IsFlagged) is cleared when content is hidden or flag is dismissed.

Normal Sequence/Flow — Viewing

  1. Admin accesses /admin/moderation.
  2. AdminModerationController checks hasModerationPermission(): allows roleId=1 or role with "Content Management" permission.
  3. Forwards to /admin/moderation.jsp. 4 tabs: Đánh Giá, Bài Viết, Bình Luận, Lịch Sử.
  4. Checkbox "Chỉ hiển thị nội dung bị báo cáo (Flagged)" for client-side filtering.
  5. Each tab loads via AJAX: reviews/posts/comments from methods with WHERE IsVisible=1, includes IsFlagged field.
  6. Each row: ID, content, author, date, status badge, flagged badge ("Bị báo cáo"), action buttons (Ẩn/Khôi phục/Bỏ qua).

Normal Sequence/Flow — Hide

  1. Admin clicks "Ẩn" → JS opens reason modal: Spam/Ngôn từ kích động/Sai lệch/Vi phạm/Khác.
  2. JS sends AJAX POST: action=Hide, entityType, entityId, reason.
  3. ModerationDAO.moderateContent() transaction:
    • UPDATE {Table} SET IsVisible = 0, IsFlagged = 0.
    • INSERT INTO ModerationRecord.

Normal Sequence/Flow — Restore

  1. Admin clicks "Khôi phục" → JS sends AJAX POST immediately: action=Restore.
  2. Sets IsVisible = 1, inserts ModerationRecord.

Normal Sequence/Flow — Dismiss Flag

  1. Admin clicks "Bỏ qua" on flagged content → AJAX POST: action=DismissFlag.
  2. Calls ModerationDAO.dismissFlag(entityType, entityId): UPDATE SET IsFlagged = 0.
