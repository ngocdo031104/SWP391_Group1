1.4 System Functionalities
1.4.1 Screens Flow
The system architecture is structured into modular sub-systems corresponding to user roles and operational domains. The flow transitions from public access (Guest) to role-specific internal dashboards upon successful authentication.
Public & Authentication Entrance: Guests interact with the Home Page, Search & Filter Tours, and Tour Details. Any attempt to create a booking, save favorites, or access community features forces a redirect to the Login / Register Account screen.
Customer Journey: Once authenticated, the Customer is routed to the Customer Dashboard. From here, they can navigate through two primary logical flows:
Transaction Flow: Tour Details $\rightarrow$ Create Booking $\rightarrow$ Process Payment (incorporating Apply Coupon) $\rightarrow$ Generate Invoice.
Post-Booking & Engagement Flow: Booking History $\rightarrow$ Request Cancellation or Submit Tour Review; and Buddy Matching $\rightarrow$ Real-Time Chat & Video Call Planning.
Back-Office Dashboards (Staff, Accountant, Guide, Admin): Upon logging in, administrative and field staff bypass public flows and land directly onto their dedicated control consoles:
Tour Guides utilize a mobile-responsive interface to check in participants, update live tour statuses, and report incidents.
Operations Staff monitor and execute bookings, manage master guest lists, assign guides, and trigger customer alerts.
Accountants supervise the financial health via transactional logs, fraud monitors, refund processing, and coupon management.
Administrators possess global overrides to configure core system parameters, manage user accounts, allocate permissions, moderate content, and view predictive business analytics.

Secondary Actors
None
Description
As a Guest or Customer, I want to view the home page so that I can see promotional banners, explore featured tours, view trending destinations, check customer testimonials, and quickly navigate to key sections of the platform.

Normal Sequence/Flow
1. The user requests to access the system's home page (via URL '/home'). 2. The system processes the request, initializes the database connection, and queries the database for: Active Tour Categories, Featured Tours (IsFeatured = 1), Popular Destinations, Top Reviews (Rating >= 4, IsVisible = 1, limit 5), and Active Coupons (IsActive = 1, start/end date valid, limit 5). 3. The page displays: Promotional banner / sliding carousel (Flash Sale Coupon), Search bar component (destinations autocomplete datalist, date, type of tour), Active categories as filter tabs, Grid of featured tours (max 9 by default, showing price, remaining capacity, title, rating. Click card to go to detail, click Wishlist button, or click "Xem thêm" to toggle list), Grid of popular destinations (max 6 by default, click card to filter tours by destination, or click "Xem thêm" to toggle list), Slider of customer testimonials (top reviews), and Newsletter subscription form in the footer.
Alternative Sequences/Flows
A1. Database Connection Failure: * If the database is down or connection times out, the servlet catches the exception, logs it, and forwards to the homepage with a friendly warning. The page loads with fallback static/placeholder values for testimonials and coupon codes. A2. Banner/Image Loading Failure: * If the system fails to load specific banner or tour images due to network lag, it will load grey placeholder graphics while keeping all textual metadata (tour titles, prices, navigation text) fully functional.

2.05.   Search and Filter Tours

Primary Actors
Guest, Customer
Secondary Actors
None
Description
As a Guest, I want to search, filter, and sort tours by destination, date, price, category, or difficulty level so that I can freely browse tour listings without making any account commitments. As a Customer, I want to search, filter, and sort tours by destination, date, price, category, or difficulty level so that I can easily find a trip that matches my specific preferences to plan a booking.
Preconditions
The user is on the tour listing page or interacting with the main search bar component. Tour metadata, departure schedules, activity status, and availability status are already initialized in the database.
Postconditions
The system displays a filtered list of tours matching the exact search and filter criteria specified by the user.
Normal Sequence/Flow
1. The user enters search parameters (destination name, date, budget) in the homepage or header search bar. 2. The server ('TourDiscoveryController.java') sanitizes the budget parameter, queries the database using 'tourDAO.searchTours(dest, null, maxPrice, date)', and retrieves the initial matching tours list and departure schedules for remaining seats. 3. The page renders search listings at '/views/tourdiscovery.jsp', displaying distinct destinations in the search datalist and active departure cities in the sidebar checkbox options. 4. The user refines their search using the sidebar filters (price range slider, categories, rating stars, duration, type, start city, seats availability) or selects a sorting option (price low-to-high, high-to-low, top-rated, newest). 5. The client-side script ('tourdiscovery.js') instantly filters and sorts the JSON data locally, updating the matching count, the tour grid, and the interactive map pins in real-time without reloading the page. 6. The user can toggle currency between USD and VND (rates are dynamically converted at 1 USD = 25,000 VND), click pins on the interactive map to view popup cards, or click the heart button to toggle wishlist status.
Alternative Sequences/Flows
A1. No Results Found: * If no tours match the user's keywords or filter combinations, the system displays a message: "No tours found matching your criteria. Please try adjusting your filters." and displays a list of recommended popular tours underneath. A2. Search Input Sanitization: * If the user enters special characters or potentially malicious scripts into the search field, the system sanitizes the input, strips out invalid characters, and returns an empty results page safely without crashing.
Business Rules
BR-21, BR-46,BR-47

Tour Deactivated: * If the selected tour is inactive or deleted from the database, the servlet automatically redirects the user back to the Home Page.

4. Non-Functional Requirements
 External Interfaces
User Interface
The system shall provide a responsive web-based interface accessible on desktop, tablet, and mobile devices.
The user interface shall support the following actors:
Admin
Staff
Customer
Guest
Accountant
Tour Guide
The interface shall provide:
Navigation menu and dashboard
Tour searching and filtering functions
Booking and payment forms
Notification center
Real-time messaging and community interaction
The interface shall follow modern UI/UX standards with consistent layouts, readable typography, and intuitive navigation.
The system shall support English language as the primary interface language.
Software Interfaces
The system shall integrate with external software/services including:
Online payment gateways for payment processing and refunds
Email service providers for account verification, notifications, and password reset
Currency exchange APIs for multi-currency conversion
Cloud storage services for tour media and images
Real-time communication services for messaging and video call planning
Analytics tools for predictive analysis and reporting
Hardware Interfaces
The system shall operate on standard cloud or on-premise servers.
End users shall access the system using:
Personal computers
Smartphones
Tablets
The system shall support standard internet-connected devices without requiring specialized hardware.
Communication Interfaces
The system shall use HTTPS protocol to ensure secure communication.
The system shall support RESTful APIs for communication between frontend and backend services.
Real-time features such as chat and notifications shall use WebSocket technology or equivalent protocols.
The system shall support secure authentication using token-based authentication mechanisms.

The system interface shall be intuitive and easy to navigate.

5. Requirement Appendix
5.1 Business Rules
ID
Rule Definition
BR-01
The system must authenticate users by comparing the submitted password after hashing with the PasswordHash stored in the User table.
BR-02
Only active and verified user accounts can access authenticated system functions.
BR-03
After successful login, the system must create a user session and redirect users according to their assigned role.
BR-04
The Remember Me feature may store only the user's email for up to 30 days and must be removed upon logout.
BR-05
Google OAuth login may automatically create a verified Customer account using Google account information.
BR-06
Registration email addresses must be unique and follow a valid email format.
BR-07
Registration passwords must contain at least 8 characters, including at least one letter and one number, and must match the confirmation password.
BR-08
Users registering an account must provide a valid full name, phone number, gender, and date of birth, and must be at least 13 years old.
BR-09
Public registration is restricted to Customer and Guide accounts only.
BR-10
User and UserProfile records must be created within the same transaction during registration.
BR-11
Newly registered accounts must be verified through OTP confirmation before becoming active.
BR-12
Only authenticated users can access and update their profile information.
BR-13
User email addresses cannot be modified through profile update functions.
BR-14
Profile updates must validate personal information and prevent invalid or malicious input.
BR-15
Avatar uploads must be image files (JPG, JPEG, PNG, GIF, or WebP) and must not exceed 2 MB.
BR-16
Password changes require verification of the current password and validation of the new password.
BR-17
The new password must differ from the current password and match the confirmation password.
BR-18
User and UserProfile information must remain synchronized after profile updates.
BR-19
Coupon or discount codes must be validated before being applied to a booking.
BR-20
Expired or inactive coupons cannot be used during payment processing.
BR-21
Multi-currency conversion rates must be updated automatically from authorized exchange-rate providers.
BR-22
Users must verify their email address before using sensitive account features such as payment or password recovery.
BR-23
Password reset requests must expire after a limited time period for security purposes.
BR-24
Only authenticated users can access the community feed, buddy matching, messaging, and review functions.
BR-25
Customers can submit tour reviews only after completing the booked tour.
BR-26
Administrators have the right to moderate or remove inappropriate reviews, comments, or community content.
BR-27
Real-time chat messages must comply with community guidelines and prohibited content policies.
BR-28
All personally identifiable information and financial data must be encrypted during transmission and storage.
BR-29
Network communications involving payment information must use secure HTTPS connections with strong encryption standards.
BR-30
The system shall maintain logs of user login activities, booking updates, and financial operations for auditing purposes.
BR-31
Revenue and operational reports can only be exported by administrators and accountants.
BR-32
Predictive analytics and business trend reports shall be generated based on historical booking and revenue data.
BR-33
A customer may maintain multiple favorite tours but cannot add the same tour to favorites more than once.
BR-34
A tour schedule cannot be deleted if there are confirmed bookings associated with it.
BR-35
Notifications related to payments, booking status, cancellations, or refunds must be automatically sent to affected users.
BR-36
Video calls and real-time communication features are available only for authenticated users participating in travel groups or buddy matching activities.
BR-37
User accounts may be temporarily locked after multiple failed login attempts for security protection.
BR-38
The system shall automatically record timestamps for all booking, payment, refund, and operational activities.
BR-39
Tour media uploaded by administrators must comply with supported file formats and size limitations defined by the system.
BR-40
The system shall retain operational and financial history records for auditing and reporting purposes according to company policy.
BR-41
The system shall display a maximum of 9 featured tours by default on the home page, with expansion toggles allowing users to view all featured tours. 
BR-42
The system shall display a maximum of 6 trending destinations by default on the home page, with expansion toggles allowing users to view all trending destinations. 
BR-43
Only active promotion coupons whose current date falls within their valid start and end dates shall be displayed in the Flash Sale section on the home page. 
BR-44
The system must validate that the entered email address follows a valid email format before accepting newsletter subscriptions from the footer section. 
BR-45
Only customer reviews that are verified, have a rating of 4 stars or higher, and are marked as visible in the database shall be displayed in the home page testimonials slider. 
BR-46
All search inputs must be trimmed, sanitized, and validated before executing database queries to prevent SQL Injection and Cross-Site Scripting (XSS) attacks. 
BR-47
Search filters including departure city, destination, category, price range, and difficulty level shall support combined filtering. If no filters are provided, all active tours shall be returned. 
BR-48
The number of remaining available seats for a departure date shall be dynamically calculated as: Schedule Capacity − Confirmed Booked Seats. 
BR-49
If a tour is deactivated by an administrator while a user is viewing it, any subsequent interaction or page refresh must redirect the user to the home page. 
BR-50
To prevent overbooking, available seats shall be temporarily reserved for 10 minutes once a customer proceeds to checkout and automatically released if payment is not completed within the reservation period. 
BR-51
Discount coupons shall not be applicable to tours that are already participating in Flash Sales or any active system-level promotional discount programs. 
BR-52
Any modification to traveler quantity or currency selection shall trigger coupon revalidation. Coupons that no longer satisfy minimum order requirements must be automatically removed. 
BR-53
The tour price and all applied discount details shall remain fixed for 15 minutes after checkout begins to protect customers from price changes during the transaction process. 
BR-54
Departure dates that have already passed shall be automatically hidden from the booking date picker and must not accept new bookings. 
BR-55
The system shall prevent logged-in users from creating duplicate bookings for the same tour and departure date when an existing booking is in Pending or Paid status. 
BR-56
When updating tour inclusions, the system shall execute a single database transaction that removes all existing inclusion records and inserts the new inclusion list to maintain data consistency. 
BR-57
The system shall automatically parse itinerary content, extract day numbers and titles, and assign appropriate Lucide icons based on predefined keyword matching rules. 
BR-58
If latitude or longitude values are empty, invalid, or non-numeric, the system shall sanitize the coordinates by assigning a fallback value of 0.0 or NULL instead of generating an error. 
BR-59
Administrators shall not be allowed to modify departure dates, base prices, or deactivate schedules that already have at least one active booking in Pending or Paid status. 
BR-60
The system shall prevent saving a tour when the number of itinerary day entries does not match the tour duration specified in the tour profile.
